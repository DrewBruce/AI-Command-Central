Executive summary

The decision is not S3 versus Clarity. It is how to avoid rebuilding downstream data consumers when DC changes the pipe.

Recommendation

Adopt a Suncorp canonical SRI acquisition layer

Suncorp should decouple downstream consumption from DC's delivery format. The durable asset should be a Suncorp-controlled SRI model with adapters from S3 parquet, Clarity Silver, and any approved exception source.

Tactical bridge

Use S3 only as a short-lived source adapter

Direct S3 Bronze parquet is the best near-term route if SRI tables reach general availability, but it should not become the downstream contract because DC intends to retire the option at end-2027.

CFO-readable problem statement

Suncorp needs reinsurance information soon enough to support business use, but the supplier's required future product path may not contain the detail Suncorp needs. The architecture should let Suncorp take the earliest viable feed without paying for repeated downstream rebuilds when the supplier changes the data product.

Default

Canonical model

Fact base and absence sweep

What is resolved, what is open, and what must be verified before funding a path.

Vendor roadmap items are from the supplied working session and must be verified with DC because capability, product dates, licensing, and retirement commitments may have changed.

Resolved facts from the session

  • S3 parquet and Clarity Bronze are fed by the same pipe: Oracle redo logs to CDC to parquet to S3 to Snowflake to BRONZE_INPUT.
  • General availability means a standard product feature of DC SaaS, not a customer-specific build.
  • S3 is independent of Clarity as a consumption route, while Clarity is already live for other DC AD DCRM customers.
  • DC intends to retire S3 at end-2027, with Clarity as the mandated end state.
  • Via Clarity, DC will expose Silver only, not Bronze.
  • Prior DI experience indicates Silver did not contain all data Suncorp required, forcing Bronze consumption.

Unresolved core issues

  • Layer mismatch: the mandated end state is the historically weaker layer for Suncorp's detailed data needs.
  • SRI objects are not yet GA anywhere, so every path depends on DC productising them.
  • Model differences across S3 parquet, Bronze Input, and Silver can cascade rework into CDP and downstream.
  • S3's usable window may be too short if GA lands around December 2026 and retirement remains end-2027.

Recommendation with switch conditions

Default to Option 4. Switch only if DC proves that Clarity Silver has complete SRI field coverage by a committed date, or if S3 SRI GA slips so late that the remaining pre-retirement window is not worth funding.

  • Switch toward Clarity-first if Silver samples prove complete field coverage, lineage, timeliness, and support.
  • Switch away from S3 bridge if S3 SRI GA is later than the point where dual-build cost exceeds business benefit.
  • Reopen Oracle mirror only if DC confirms CDC support, contractual access, and operational ownership.
Recommended solution design

A replaceable-source architecture, not a source-shaped downstream architecture.

The logical architecture deliberately keeps DC source formats outside the enduring downstream contract.

Principle 1

Preserve downstream stability

Prove field coverage before cutover

Sacrifice: Clarity cannot be accepted as end state on roadmap promise alone.

Dual-run before source retirement

Sacrifice: temporary operating cost. Benefit: avoids blind cutover into a lossy model.

Near-term S3 flow

  1. DC publishes SRI tables to S3 parquet under a productised GA commitment.
  2. Suncorp S3 adapter reads source files and captures schema/version metadata.
  3. Adapter maps records into the canonical SRI acquisition model.
  4. Quality controls reconcile record counts, mandatory fields, reference values, and late-arriving changes.
  5. CDP publishes the stable SRI contract to downstream consumers.

End-state Clarity flow

  1. DC provides Clarity Silver SRI samples, lineage, and field definitions.
  2. Suncorp compares Silver against required canonical SRI fields and S3 output.
  3. Clarity adapter maps Silver into the same canonical SRI model.
  4. S3 and Clarity run in parallel until reconciliation thresholds are met.
  5. Architecture authority approves source cutover and controlled S3 retirement.
Cutover should be based on data equivalence and operational readiness, not on the calendar alone.

DC confirmation pack

The decision should be gated by written vendor evidence, not roadmap interpretation.

Productisation and dates

  • Will SRI tables reach GA in S3, and on what committed date?
  • Will the same SRI objects reach GA in Clarity Silver, and on what committed date?
  • Is end-2027 S3 retirement hard, or can S3 persist under exception?

Data content and lineage

  • Provide field-level samples for S3 parquet and Clarity Silver.
  • Provide lineage from operational object to Silver field.
  • Confirm whether Silver contains all Suncorp-required SRI fields and history.

Technical and commercial viability

  • How is data reaching Clarity today if S3 is not GA until December?
  • Is mirrored Oracle RDS with CDC technically supported and contractually permitted?
  • What are support, access, licensing, security, and service limits for each path?

Success metrics

  • 100% of mandatory SRI fields mapped into canonical model.
  • Record-count reconciliation within agreed tolerance for each load.
  • Zero downstream consumers bound directly to S3 parquet or Clarity Silver structures.
  • Schema change impact contained to adapter and canonical model governance.
  • Dual-run exit achieved before S3 retirement or extension decision point.
First move

Lock the SRI field contract

Before debating source choice, define the Suncorp-required SRI data contract and mandatory reconciliation rules.

Second move

Demand DC evidence

Ask for written GA timelines, sample data, lineage, access constraints, and retirement commitments.

Third move

Prototype the adapter

Build the smallest source adapter needed to prove the canonical model and quality controls.
